# 🏗️ 工程设计层规范
> 定位：全链路决策闸门｜工程稳健优先｜风险前置｜适度设计

## 1. 顶层核心准则
- **工程稳健优先**：功能可靠、可维护、易迭代为第一目标
- **架构适度设计**：按需抽象，拒绝预设计、拒绝无效解耦
- **长期可迭代**：模块低耦合、依赖清晰、职责边界明确
- **团队可读性**：代码写给人读，逻辑线性、意图直观
- **风险前置**：参数校验、错误捕获、资源回收标准化
- **版本兼容稳定**：迭代不破坏已有接口、存量逻辑、旧配置

## 2. 需求前置三问（通用工程准则）
- [ ] 该需求是否为真实业务/生产问题？（拒绝假想优化）
- [ ] 是否存在轻量化、低改造的替代方案？（拒绝大炮打蚊子）
- [ ] 迭代改动是否影响存量功能与兼容性？（拒绝隐性破坏）

## 3. 五层拆解分析框架
| 层级 | 核心焦点 | 管控规则 |
|:---|:---|:---|
| **L1 数据结构与流转** | 核心实体、数据关系、归属权、修改链路 | 杜绝无意义拷贝、类型迂回转换 |
| **L2 分支与边界逻辑** | 全部分支条件、异常分支、临时补丁逻辑 | 区分原生设计缺陷与业务必要判断 |
| **L3 逻辑复杂度管控** | 核心功能一句话收敛定义 | 控制层级/嵌套/概念数量，减少认知负担 |
| **L4 变更破坏性评估** | 接口、依赖、存量逻辑、配置、下游影响 | 最小改动范围，隔离变更域 |
| **L5 落地实用性校验** | 场景真实度、受众覆盖、问题严重程度 | 方案复杂度与问题量级严格匹配 |

## 4. 决策输出模板
```text
【核心判断】
✅ 值得做：[原因：业务价值/技术收益/风险可控]
❌ 不值得做：[原因：伪需求/过度设计/兼容性风险]

【关键洞察】
- 数据结构：核心关联与设计短板
- 复杂度：可精简/可合并/可移除冗余点
- 风险点：[兼容风险, 逻辑风险, 维护风险]

【标准化落地方案】
1. 收敛数据模型，统一数据流转
2. 精简分支逻辑，合并重复判断
3. 线性化执行流程，降低嵌套层级
4. 隔离变更，保证存量兼容稳定
```
