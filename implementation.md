# ✍️ 编码层规范
> 定位：日常开发执行标尺｜优势固化｜短板拦截｜务实落地

## 1. 固有优势固化（✅ 保持并强化）
- **模块化拆分**：目录结构规整，按业务域划分，代码解耦增强复用性
- **命名规范**：标识符语义化、表意明确，拒绝混乱简写；坚持驼峰命名，公共函数首字母大写，私有函数首字母小写
- **格式统一**：基础代码格式、缩进、留白统一，可读性基础良好；一切函数必须通过公有函数调用才能公开，保证安全性
- **务实开发**：功能落地优先，不盲目堆砌花哨语法，聚焦问题关键，关注解决核心问题
- **Git 纪律**：提交克制，单提交诉求明确，无垃圾杂糅提交；保证每次提交是一个独立模块的修改/删除/增加
- **技术栈聚焦**：以 Go + Vue3 为主，擅长 Go/前端/运维多场景，工程落地意识强

## 2. 优化短板专项约束（⚠️ 重点管控）
| 短板场景 | 管控规则 | 落地范式 |
|:---|:---|:---|
| **函数职责膨胀** | 单一职责贯彻不彻底 | 核心功能必须能用一句话定义；超限强制拆分为 `校验→转换→执行` |
| **分支补丁式判断** | 边界场景依赖多层 If | 同类判断≥2次 → 提取策略函数/配置表；优先重构数据结构消除特例 |
| **注释质量偏差** | 偏向功能描述，缺动机 | 仅补充设计动机、特殊场景、兼容原因；禁止复述代码行为 |
| **嵌套深度超标** | 局部逻辑缩进易突破 3 层 | 硬性限制：嵌套 ≤ 3 层；超限视为设计瑕疵，需提前卫语句或策略抽离 |
| **异常拦截零散** | 参数校验无统一规范 | 入口层统一拦截：参数校验 → 错误捕获 → 资源回收/日志上报标准化 |
| **过度抽象倾向** | 提前预留无用设计 | 拒绝为假想问题增加抽象/拦截/限制；代码服务现实业务，不服务炫技 |

## 3. 最终统一编码规范（执行底线）
- 以工程稳健、长期可维护为顶层核心
- 逻辑线性化，严控嵌套与复杂分支
- 数据结构优先优化，减少上层逻辑补丁
- 全场景基础防护：入参校验 + 错误捕获
- 命名直白克制，表意优先
- 兼容优先，存量行为不随意变更
- **Linus 简洁哲学仅作为 Review 收紧标尺，不主导架构设计**
