# 🔍 Review 评审层规范
> 定位：代码质量守门员｜四层过滤｜Linus 品味专项｜客观收敛

## 1. 四层评审体系
1. **基础工程规范**（主体准则）：命名、格式、错误处理、提交原子性（硬性 100% 达标）
2. **业务逻辑正确性**：需求覆盖完整、边界场景处理、数据一致性（用例驱动评审）
3. **性能与资源管控**：N+1 查询、内存泄漏风险、重复计算、缓存策略（需 Profiling 数据支撑）
4. **Linus 风格专项评审层**（附加约束）：仅作用于代码风格、复杂度、设计品味审查

## 2. Linus 评审层核心规则（仅 Review 生效）
- **🎨 Good Taste 品味评审**：优先通过重构数据结构消除特殊边界；减少补丁式 If 分支；优秀的实现应当消灭特例，而非不断兼容特例
- **📏 简洁性强制审查**：缩进嵌套禁止长期超过 3 层（超限视为设计瑕疵）；函数短小单一，拒绝大函数堆砌；复杂性是维护隐患，无理由复杂必被质疑
- **⚙️ 纯粹实用主义评审**：拒绝理论完美但落地臃肿的设计；不为假想问题增加抽象；代码服务现实业务
- **🛡️ 兼容底线补充约束**：隐性规则：拒绝无意识破坏存量行为；所有隐性行为变更，必须显式评估与标注

## 3. 标准化评分与结论输出
```text
【代码品味评分】
🟢 合格优雅 / 🟡 勉强可用 / 🔴 设计瑕疵

【致命缺陷】
- [列出逻辑、结构、风格、复杂度核心问题]

【优化指引（Linus 风格建议）】
- 🔹 重构结构以消除多余分支
- 🔹 压缩冗余逻辑，多行逻辑收敛为极简实现
- 🔹 调整数据模型，从根源减少判断
```

## 4. 强制检查清单（PR 合并门槛）
- [ ] 命名规范、语义完整，无混乱简写
- [ ] 函数职责单一、长度可控
- [ ] 嵌套层级合规（≤3），无深度堆叠
- [ ] 分支逻辑精简，无补丁式冗余判断
- [ ] 注释只补充设计动机与特殊场景
- [ ] 错误处理、参数校验完整
- [ ] 无过度抽象、无无效预留设计
- [ ] 提交原子化，单次变更诉求清晰

## 5. 沟通与评审表达规范
- **表达**：直接、客观、零废话
- **批判**：仅针对代码设计、逻辑、风格，不针对个人
- **评审原则**：先保证工程稳定，再用 Linus 品味做精简优化
